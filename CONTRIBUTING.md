# راهنمای مشارکت در JalaliDatePicker

از اینکه می‌خواهید به بهبود **JalaliDatePicker** کمک کنید ممنونیم. این راهنما توضیح می‌دهد چطور باگ گزارش کنید، قابلیت جدید پیشنهاد دهید، کد بفرستید و تغییرات را قبل از ارسال بررسی کنید.

لطفاً هنگام مشارکت، [Code of Conduct](CODE_OF_CONDUCT.md) پروژه را رعایت کنید.

## گزارش باگ

قبل از ثبت باگ جدید، ابتدا [issueهای باز](https://github.com/majidh1/JalaliDatePicker/issues) را بررسی کنید تا مورد مشابهی وجود نداشته باشد.

برای گزارش باگ، از قالب **گزارش باگ** در GitHub Issues استفاده کنید و تا حد امکان این اطلاعات را وارد کنید:

- نسخه JalaliDatePicker
- مرورگر و سیستم‌عامل
- تنظیمات استفاده‌شده برای `jalaliDatepicker.startWatch`
- مراحل دقیق بازتولید مشکل
- نتیجه مورد انتظار و نتیجه فعلی
- نمونه کد، لینک demo، screenshot یا خطای console در صورت وجود

گزارش‌هایی که نمونه قابل بازتولید دارند سریع‌تر بررسی می‌شوند.

## پیشنهاد قابلیت جدید

برای پیشنهاد قابلیت جدید، از قالب **درخواست قابلیت** در GitHub Issues استفاده کنید.

لطفاً توضیح دهید:

- چه نیازی دارید یا چه مشکلی را می‌خواهید حل کنید
- API پیشنهادی شما چیست
- آیا راه‌حل موقت یا جایگزین فعلی وجود دارد
- این قابلیت برای چه نوع پروژه‌هایی مفید است

برای حفظ سادگی کتابخانه، قابلیت‌های جدید باید API کوچک، روشن و سازگار با رفتار فعلی داشته باشند.

## راه‌اندازی محیط توسعه

ابتدا repository را fork و clone کنید، سپس dependencyها را نصب کنید:

```shell
npm install
```

اجرای محیط توسعه:

```shell
npm run dev
```

این دستور Vite را اجرا می‌کند و صفحه نمونه `test/index.html` را باز می‌کند.

## بررسی تغییرات قبل از Pull Request

قبل از ارسال Pull Request این دستورها را اجرا کنید:

```shell
npm test
npm run lint
npm run build
```

انتظار داریم هر سه دستور بدون خطا اجرا شوند.

## ساختار خروجی

فایل‌های قابل انتشار در پوشه `dist` ساخته می‌شوند:

```text
dist/jalalidatepicker.css
dist/jalalidatepicker.js
dist/jalalidatepicker.min.css
dist/jalalidatepicker.min.js
```

اگر تغییری در سورس، استایل یا build می‌دهید، `npm run build` را اجرا کنید تا فایل‌های `dist` هم به‌روز شوند.

## راهنمای کدنویسی

- تغییرات را تا حد امکان کوچک و متمرکز نگه دارید.
- از الگوهای موجود در کد پروژه پیروی کنید.
- برای تغییرات رفتاری، تست اضافه یا به‌روزرسانی کنید.
- APIهای عمومی را بدون دلیل قوی تغییر ندهید.
- اگر behavior قدیمی را تغییر می‌دهید، در Pull Request توضیح دهید چرا این تغییر لازم است.
- مستندات فارسی و انگلیسی را با هم به‌روز کنید، مخصوصاً اگر option یا attribute جدید اضافه می‌شود.

## Pull Request

برای ارسال Pull Request:

1. یک branch جدا بسازید.
2. تغییرات را اعمال کنید.
3. تست، lint و build را اجرا کنید.
4. توضیح کوتاه و روشن برای Pull Request بنویسید.
5. issue مرتبط را لینک کنید، مثلاً `Closes #153`.

یک Pull Request خوب معمولاً شامل این موارد است:

- خلاصه تغییرات
- دلیل تغییر
- روش تست
- screenshot یا demo برای تغییرات UI، اگر لازم باشد

## نسخه‌دهی و انتشار

انتشار npm توسط maintainer انجام می‌شود. workflow انتشار فقط زمانی publish می‌کند که نسخه موجود در `package.json` با نسخه منتشرشده در npm متفاوت باشد.

برای release، maintainer باید:

1. نسخه `package.json` را به‌روزرسانی کند.
2. changelog را به‌روزرسانی کند.
3. `npm test`, `npm run lint`, و `npm run build` را اجرا کند.
4. تغییرات را در branch اصلی merge کند.
